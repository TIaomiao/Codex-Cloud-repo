## 今日摘要

- Codex 正在从本机 coding 助手变成跨设备、可远程接力、可长期运行的工作代理。
- 权限、额度、远控和审计边界会越来越影响 Codex 是否能从个人试用扩展到团队工作流。
- GitHub 开始把 Copilot 采用情况按 agent 阶段分层，这比只看活跃人数更适合衡量 AI 工作流成熟度。
- MCP 调试工具继续成熟，但新工具对 Node 22+ 等运行时基线提出了更高要求。

## 动态列表

### 1. OpenAI 把 Windows Computer Use、远程接管和使用画像放进 Codex 主线

- 分类：Codex
- 来源：[OpenAI Help Center](https://help.openai.com/en/articles/6825453-chatgpt-versionshinweise)
- 发生了什么：OpenAI 的版本说明把 Windows Computer Use、远程接管和 Codex 使用画像放进 Codex 相关更新中，显示 Codex 正在朝跨设备、远程主机和长期任务接力发展。
- 为什么重要：这让 Codex 更像长期运行在主机上的工作代理，而不是只能在本机前盯着的 coding 助手。对需要守着桌面应用、浏览器或长任务队列的场景，工作流设计会明显变化。
- 我可以怎么用：挑一个原本必须守在 Windows 主机旁边的长任务做试点，例如桌面复现、浏览器回归或后台脚本盯跑，把“主机执行、移动端接力决策”当成新的任务拆分方式。

### 2. OpenAI 更新 Codex 计划说明，权限和额度边界更明确

- 分类：Codex
- 来源：[OpenAI Help Center](https://help.openai.com/en/articles/11369540)
- 发生了什么：OpenAI 更新 Codex 计划说明，让不同计划下的权限、额度和使用边界更清楚。
- 为什么重要：这条更新直接影响团队是否能把 Codex 从个人试用扩展到团队常规工作流。很多 rollout 风险不在模型本身，而在额度、权限和审计边界不清楚。
- 我可以怎么用：按“本地使用、云端委托、跨端远控、插件调用”四类方式梳理团队权限和预算；没有这层治理前，不要把更多自动化任务默认接入 Codex。

### 3. GitHub 开始把 Copilot 采用情况按 agent 阶段分层

- 分类：Workflow
- 来源：[GitHub Changelog](https://github.blog/changelog/2026-05-29-copilot-usage-metrics-api-adds-cohorts-for-ai-adoption)
- 发生了什么：GitHub Copilot usage metrics API 增加 AI adoption cohorts，用阶段化方式描述团队采用 AI/Copilot 的深度。
- 为什么重要：这让团队不再只能看“有没有人在用 AI”，而能看出大家是否真的从代码补全迁移到了 cloud agent、CLI、code review 等 agent 工作流。对培训、预算和推广复盘都更有解释力。
- 我可以怎么用：内部复盘时补上 adoption phase 维度，至少区分代码优先、单 agent 和多 agent；后续把案例沉淀和 enablement 资源重点投向 Agent first 到 Multi-agent 的转化。

### 4. mcp-debugger v0.21.0 继续推进结构化 agent 调试，但要求 Node 22+

- 分类：MCP
- 来源：[debugmcpdev/mcp-debugger](https://github.com/debugmcpdev/mcp-debugger)
- 发生了什么：mcp-debugger 发布 v0.21.0，继续推进面向 MCP/agent 的结构化调试能力，同时将最低运行时要求提高到 Node.js 22+。
- 为什么重要：它代表一类更成熟的 coding-agent 工具形态：不再只靠日志和 shell 输出排错，而是把调试能力结构化成可调用工具。代价是本地环境和 runner 镜像需要跟进 Node 22 基线。
- 我可以怎么用：如果你在做本地 coding agent 工具栈，先在隔离环境试一条最小调试链路，再决定是否把这类工具接入默认开发容器或 CI runner。
